Review task: Focus on avoiding bad comments. Only issue suggestions that are related to avoiding useless verbose comments. Comments should only be added when they add significant value to the code, which typically happens when they explain *why* something is structured a certain way.

Examples of BAD comments:

    import os  # Added import

    # Iterate the list of elements
    for i in elements:
      ...

Commented out code should generally be removed.
An exception to that could be if the commented-out code
helps explain the code
(for example, providing some pseudo-code or such).

Examples of GOOD comments:

    class Foo {
      private offsetX: number;   // X-offset for panning the canvas.
      private offsetY: number;   // Y-offset for panning the canvas.
      …
    }

These are good comments (even if they are somewhat obvious) because good
documentation about the semantics of class attributes (data) can help make
software significantly more maintainable.
